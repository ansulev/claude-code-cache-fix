# Claude Code v2.1.97 Cache Behavior Test Report

**Date:** 2026-04-09
**Tested by:** VSITS LLC / Chris Nighswonger
**Interceptor version:** claude-code-cache-fix v1.5.1

## Summary

v2.1.97 still exhibits the resume cache regression. On `--continue` (resume), Claude Code creates ~5,200 tokens of unnecessary cache per turn due to block scatter in `messages[0]`. The `claude-code-cache-fix` interceptor eliminates this entirely — zero cache creation across all resume turns.

## Test Methodology

Four scenarios tested via automated script, each capturing cache stats from the Anthropic API response:

1. **One-shot WITHOUT interceptor** — two sequential `-p` calls
2. **One-shot WITH interceptor** — same, with `NODE_OPTIONS="--import claude-code-cache-fix"`
3. **Multi-turn + resume WITHOUT interceptor** — named session, 3 turns using `--continue`
4. **Multi-turn + resume WITH interceptor** — same session pattern

Cache metrics captured from `quota-status.json` after each API call: TTL tier, `cache_creation_input_tokens`, `cache_read_input_tokens`, `ephemeral_1h_input_tokens`, `ephemeral_5m_input_tokens`.

## Results

### One-shot (Tests 1-2): No difference

| Call | Interceptor | Create | Read | TTL | Hit Rate |
|------|-------------|-------:|-----:|-----|----------|
| 1a | No | 0 | 26,145 | 1h | 100% |
| 1b | No | 0 | 26,145 | 1h | 100% |
| 2a | Yes | 0 | 26,145 | 1h | 100% |
| 2b | Yes | 0 | 26,145 | 1h | 100% |

One-shot calls hit warm cache from prior session activity. No cache regression in one-shot mode.

### Multi-turn + Resume (Tests 3-4): Interceptor eliminates cache waste

| Turn | Interceptor | Create | Read | TTL | Hit Rate | Interceptor Action |
|------|-------------|-------:|-----:|-----|----------|--------------------|
| 3-T1 | **No** | **5,204** | 20,939 | 1h | 80.1% | — |
| 3-T2 | **No** | **5,220** | 20,939 | 1h | 80.0% | — |
| 3-T3 | **No** | 18 | 26,159 | 1h | 99.9% | — |
| 4-T1 | Yes | **0** | 26,143 | 1h | **100%** | — |
| 4-T2 | Yes | **0** | 26,159 | 1h | **100%** | Relocation applied |
| 4-T3 | Yes | **0** | 26,177 | 1h | **100%** | Relocation applied |

**Without interceptor:** Turns 1-2 after resume waste ~5,200 tokens each in `cache_creation` due to attachment block scatter. Turn 3 finally stabilizes at 99.9%. Total waste: ~10,400 tokens across two turns.

**With interceptor:** Zero `cache_creation` on all turns. The interceptor detected and relocated scattered blocks on turns 2-3, maintaining a stable prefix. 100% cache hit from the first turn.

## What This Means for Cost

At Sonnet 4.5 rates:
- Cache creation (1h): $6.00 / MTok
- Cache read: $0.30 / MTok

Per resume session without the interceptor, the ~10,400 wasted tokens cost an additional ~$0.06 in cache creation that should have been cache reads. This compounds across sessions — a user resuming 10 times per day wastes ~$0.60/day in unnecessary cache rebuilds. On Opus at 2x the rates, that doubles.

This doesn't account for the larger cost: when the cache prefix is unstable, the *entire* cached prefix is rebuilt, not just the scattered portion. The 5,200 tokens of creation represent the delta; the full rebuild cost depends on system prompt + tools size (~26K tokens in our test).

## Conclusion

**v2.1.97 has not fixed the resume cache regression.** The block scatter bug documented in [#34629](https://github.com/anthropics/claude-code/issues/34629), [#44045](https://github.com/anthropics/claude-code/issues/44045), and [#41930](https://github.com/anthropics/claude-code/issues/41930) is still present.

The `claude-code-cache-fix` interceptor ([npm](https://www.npmjs.com/package/claude-code-cache-fix) / [GitHub](https://github.com/cnighswonger/claude-code-cache-fix)) eliminates the regression entirely. Install with:

```bash
npm install -g claude-code-cache-fix
```
