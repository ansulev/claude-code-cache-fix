# Claude Code v2.1.97 — Compatibility Test Report

**Date:** 2026-04-09
**Tested by:** VSITS LLC / Chris Nighswonger
**Interceptor version:** claude-code-cache-fix v1.5.1

## Summary

We tested Claude Code v2.1.97 to assess which known cache and context issues remain, and how the `claude-code-cache-fix` interceptor performs against the current release. The resume cache regression is still present. Image carry-forward, tool reorder instability, and the lack of cost visibility also remain unaddressed. The interceptor eliminates or mitigates all of these.

## 1. Resume Cache Regression

### Methodology

Four scenarios tested via automated script (`tools/cache-test.sh`), each capturing cache stats from the Anthropic API response:

1. **One-shot WITHOUT interceptor** — two sequential `-p` calls
2. **One-shot WITH interceptor** — same, with `NODE_OPTIONS="--import claude-code-cache-fix"`
3. **Multi-turn + resume WITHOUT interceptor** — 3 turns using `--continue`
4. **Multi-turn + resume WITH interceptor** — same pattern

### Results: One-shot (Tests 1-2) — No difference

| Call | Interceptor | Create | Read | TTL | Hit Rate |
|------|-------------|-------:|-----:|-----|----------|
| 1a | No | 0 | 26,145 | 1h | 100% |
| 1b | No | 0 | 26,145 | 1h | 100% |
| 2a | Yes | 0 | 26,145 | 1h | 100% |
| 2b | Yes | 0 | 26,145 | 1h | 100% |

One-shot calls hit warm cache. No regression in one-shot mode.

### Results: Resume (Tests 3-4) — Bug still present

| Turn | Interceptor | Create | Read | TTL | Hit Rate | Interceptor Action |
|------|-------------|-------:|-----:|-----|----------|--------------------|
| 3-T1 | **No** | **5,204** | 20,939 | 1h | 80.1% | — |
| 3-T2 | **No** | **5,220** | 20,939 | 1h | 80.0% | — |
| 3-T3 | **No** | 18 | 26,159 | 1h | 99.9% | — |
| 4-T1 | Yes | **0** | 26,143 | 1h | **100%** | — |
| 4-T2 | Yes | **0** | 26,159 | 1h | **100%** | Relocation applied |
| 4-T3 | Yes | **0** | 26,177 | 1h | **100%** | Relocation applied |

**Without interceptor:** Turns 1-2 after resume waste ~5,200 tokens each in `cache_creation` due to attachment block scatter. Turn 3 stabilizes. Total waste: ~10,400 tokens across two turns.

**With interceptor:** Zero `cache_creation` on all turns. 100% cache hit from turn 1.

### Cost impact (resume only)

At Sonnet 4.5 rates (1h cache creation: $6.00/MTok, cache read: $0.30/MTok):
- Per resume: ~$0.06 wasted in cache creation
- 10 resumes/day: ~$0.60/day
- On Opus (2x rates): ~$1.20/day

## 2. Image Carry-Forward

### The problem

Images read via the Read tool are encoded as base64 in `tool_result` blocks and persist in conversation history. They are sent on **every subsequent API call** until compaction. Claude Code v2.1.97 does not strip or age-out these images.

### Measured impact (real-world session, v2.1.96–2.1.97)

Data from an image-intensive coding session using Code Agent:

| Metric | Value |
|--------|-------|
| Images accumulated | 77 |
| Total image events (images × turns carried) | 219 |
| Tokens consumed by image carry-forward | 31.1M |
| Per-image cost per turn | ~62,500 tokens |
| Projected daily cost at Opus rates ($5/MTok input) | **$155.66** |

A single 500KB image adds ~62,500 tokens to every subsequent API call. In a session with many Read calls on screenshots or diagrams, this dominates all other cost factors.

### Interceptor fix

`CACHE_FIX_IMAGE_KEEP_LAST=N` strips base64 images from `tool_result` blocks older than the last N user messages. Only targets images inside tool results — user-pasted images are never touched. Files remain on disk for re-reading.

```bash
export CACHE_FIX_IMAGE_KEEP_LAST=3
```

**This is the interceptor's highest-impact feature.** In the session measured above, image stripping would have eliminated the vast majority of the 31.1M carry-forward tokens.

Credit: The base64 image carry-forward bug was first identified by [@Renvect](https://github.com/Renvect) in [#40524](https://github.com/anthropics/claude-code/issues/40524).

## 3. Skills and Deferred Tools Instability

### The problem

Skills (loaded from filesystem) and deferred tools listings can vary between API calls due to:
- **Non-deterministic `readdir` ordering** — filesystem enumeration doesn't guarantee order
- **Trailing whitespace jitter** — serialization produces 1-char differences between calls

Both change the content of `messages[0]`, breaking the cache prefix even when no blocks have scattered.

### Confirmed in v2.1.97

This was reported by [@bilby91](https://github.com/bilby91) on [#44045](https://github.com/anthropics/claude-code/issues/44045) using the Agent SDK with MCP tools. Debug trace showed:
- Skills block: same size (3,951 chars) but different entry order between calls
- Deferred tools block: 754 → 755 chars (1-character whitespace difference)

### Interceptor fix (v1.5.1)

- `sortSkillsBlock()` — alphabetical sort of skill entries
- `sortDeferredToolsBlock()` — alphabetical sort of deferred tool names
- `pinBlockContent()` — whitespace normalization + content hash pinning

These apply even when blocks stay in `messages[0]` (non-scattered path), covering both the resume scatter case and the content jitter case.

## 4. Cost Visibility

### The problem

Claude Code provides no per-call cost breakdown, no cache efficiency metrics, and no way to attribute costs to specific sessions or time windows. Users see quota percentages but not dollars. API users see bills but can't trace them to specific activity.

### Interceptor solution (v1.5.0+)

**Usage telemetry:** The interceptor logs one JSON line per API call to `~/.claude/usage.jsonl`:

```json
{"timestamp":"2026-04-09T12:00:00Z","model":"claude-sonnet-4-5-20250929","input_tokens":50000,"output_tokens":1200,"cache_read_input_tokens":13000,"cache_creation_input_tokens":0,"ephemeral_1h_input_tokens":0,"ephemeral_5m_input_tokens":0,"ttl_tier":"1h"}
```

**Cost report tool:** Calculates costs from telemetry with three pricing sources:

```bash
node tools/cost-report.mjs                    # from interceptor log
node tools/cost-report.mjs --date 2026-04-08  # specific date
node tools/cost-report.mjs --admin-key <key>  # cross-reference with Admin API
node tools/cost-report.mjs --sim-log sim.log  # from simulation logs
```

Supports text, JSON, and Markdown output. Validated against the Anthropic Admin API with $0.00 delta on an 18-call test run.

Also works standalone for SDK and proxy users via `--file` or stdin input.

## 5. TTL Tier and Quota Tracking

### The problem

Claude Code does not expose which cache TTL tier the server assigned (1h vs 5m), what your current quota utilization is, or whether you're in a peak hour window. Users discover TTL downgrades only after the damage is done.

### Interceptor solution

On every API call, the interceptor:
- Clones the SSE response stream and extracts `ephemeral_1h_input_tokens` and `ephemeral_5m_input_tokens` to determine TTL tier
- Parses response headers for `anthropic-ratelimit-unified-5h-utilization` and `7d-utilization`
- Detects Anthropic peak hours (weekday 13:00-19:00 UTC)
- Writes all of this to `~/.claude/quota-status.json` for consumption by status line hooks or dashboards
